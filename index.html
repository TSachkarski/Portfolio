Why it exists
Ambient Motion

A background that responds to presence creates life without demanding attention. The particles don't interrupt the content — they reward curiosity.

The physics
Repulsion + Friction

Each particle carries velocity damped by a 0.96 friction coefficient per frame. The cursor applies a force inversely proportional to distance — close particles scatter fast, far ones drift.

The constraint
Zero Dependencies

No Three.js, no GSAP, no WebGL. Vanilla canvas 2D API with devicePixelRatio scaling for retina sharpness. The constraint is the point.

Reflection

THE RULE BEHIND
EVERY ANIMATION.

3s
Max animation duration

Anything longer than 3 seconds stops being animation and starts being load time. Keep every motion under 1 second where possible — 3s is the absolute ceiling.

1
Job per animation

Signal, confirm, guide, or delight. Pick one. An animation trying to do all four does none of them well. When you can't explain what one job an animation does, remove it.

0
Animations without purpose

If you can't complete the sentence "this animation exists to ___", it shouldn't exist. Motion is a communication tool. When it communicates nothing, it is noise.

01 — The Problem

AGENTS HAD NOTHING.

Contact center agents are under constant cognitive load. They handle live calls, resolve tickets, manage their schedule, and track their own performance — often simultaneously. The tools built to support them are almost universally the opposite of helpful.

  • Performance data buried in weekly manager reports — no real-time KPI visibility
  • Schedule in a separate system — agents context-switched between 3–4 tools to understand their day
  • Alerts via email or chat — asynchronous, disconnected from the moment they mattered
  • Mobile experience: non-existent
  • Managers had dashboards. Agents had nothing.

"How do you give an agent enough information to self-manage their performance — without adding to an already overwhelming cognitive environment? And how do you make that work for someone mid-call, on a small screen, with 15 seconds to glance at it?"

02 — My Role

SOLE DESIGNER. SOLE DEVELOPER.

Every decision — from information architecture to animation easing curves — was mine to make and justify. I designed and built the full prototype with no handoff point: design and development happened in the same session.

Design
What I Designed

Information architecture across 5 screens. Full design system: colour tokens, typography scale, component library. All interaction patterns including hover states, transitions, mobile adaptations. Full WCAG 2.1 AA audit.

Dev
What I Built

Entire prototype in a single HTML file — no frameworks, no build tools. All animations in CSS: sparklines, bar fills, card stagger, pulse rings. Responsive layout that adapts meaningfully between desktop and mobile. Semantic HTML, ARIA labels, focus management.

Early exploration — lo-fi dashboard wireframe, iteration 2
03 — The Process

HIERARCHY FIRST. SYSTEM BEFORE SCREENS.

The first question was not "what should this look like" but "what does an agent actually need to know, in what order, and how often?" I mapped information into four tiers before touching any design tool.

TierInformationGlance FrequencyDesign Response
1 — Immediate Am I on schedule right now? Every few minutes KPI cards + NOW indicator always visible
2 — Session How is today going overall? Every 30–60 min Dashboard with sparklines and alerts
3 — Weekly Am I trending correctly? Daily Performance screen with trend tables
4 — Structural What is my week, who do I report to? Occasional Schedule and Profile screens
Process artefact — Information architecture map

Phase 2: Design system before screens. I established the full token system first — colour, type, spacing — as a functional decision, not an aesthetic one. A contact center interface displays status information constantly. The colour system needed to communicate meaning instantly: Green = on target, Amber = warning, Blue = informational. All colours passed WCAG 2.1 AA at 3:1 for UI components on dark backgrounds.

Phase 4: Prototype as a design tool. Rather than static mockups, I built interactively throughout. Every motion decision (the 900ms bar animation, the 70ms card stagger delay, the spring easing on the avatar) was tested in the actual prototype — not estimated.

04 — Key Design Decisions

WHY THIS, NOT THAT.

Mobile layout exploration — "different task, not just smaller screen"
01
Sparklines Over Static Numbers

An agent seeing CSAT at 84% with a downward sparkline acts differently than one seeing 84% after two days of recovery. The trend is the signal — the number is just the current position on that trend. I chose filled area sparklines over bar charts because the area communicates volume and momentum, not just point-to-point change.

02
Ambient Awareness Over Explicit Alerts

Most dashboards solve the alert problem by adding more alerts. I took the opposite approach: encode status directly into the interface so agents absorb it without reading. The live dot pulses — you know the data is live without reading "Live." The NOW indicator is amber, not red — it marks position without implying danger.

03
Mobile as a Different Task

On desktop, an agent has time to analyse. On mobile, they're between calls with 15 seconds. The mobile layout wasn't a responsive adaptation — it was a different information strategy: stacked time blocks, icon-only navigation, 2-column KPI grid, compressed performance tables.

04
Progressive Disclosure via Overlays

Tapping any KPI card opens a detail overlay with the full week trend and stat breakdown. This keeps the dashboard clean for status checks while giving agents who want to dig deeper a path — without leaving the screen or loading a new page.

06 — Reflection

WHAT I'D DO DIFFERENTLY.

Start with real users. I designed based on first principles and secondary research. A single round of contextual inquiry — sitting with an agent for a few hours — would have surfaced edge cases I couldn't anticipate. What happens during a call surge? Do agents even look at CSAT in real time, or do they find it demoralising?

Test the information hierarchy earlier. I validated the hierarchy through logic, not testing. A card sorting session with 5 agents would have told me whether Performance and Schedule belong as separate screens, or whether agents think about them together.

Design for failure states. The prototype assumes everything is working. Real contact center software goes down, loses data connections, and shows stale numbers. I'd design explicit states for: data loading, data unavailable, connection lost, and metrics outside normal range.

"Most portfolio projects show consumer apps — food delivery, travel, e-commerce. AgentOS is different. It's enterprise UX for a genuinely difficult problem: information density, real-time data, human-AI collaboration, and the constraint that the user might be mid-call. Designing for that environment — and shipping a working, accessible, animated prototype — is the work."

01 — The Problem

LUXURY BRANDS THAT BETRAY THEMSELVES.

I spent a week auditing 12 furniture and homeware brands across the premium tier — Muuto, HAY, The Future Perfect, Aesop, Vipp, and others. Three failure modes came up repeatedly.

P1
Generic UI Kills Brand Trust

Most mid-tier furniture sites use off-the-shelf UI patterns that could belong to any category. A chair that costs €1,200 is presented with the same visual weight as a €12 phone case. Speed over intentionality — templates are fast, custom design systems are slow.

P2
The Checkout Betrayal

Even on brands that get the editorial feel right, the moment you click "Add to Cart" the experience collapses. You're ripped out of context, thrown into a clinical transaction UI, and the mood is broken. A luxury brand cannot afford to feel like a transaction.

P3
Accessibility as Afterthought

Almost every luxury site I audited had WCAG AA contrast failures — particularly on muted secondary text, which is ironic because that's exactly the text they use most. Focus indicators were invisible. Screen reader structure was absent.

Goal
My Design Constraints

Bring the quiet confidence of a physical furniture showroom into a digital experience — without losing the feeling. Keep the brand coherent from homepage to checkout. Pass WCAG AA on every text pair, from day one.

02 — Process

SYSTEM BEFORE SCREENS. ALWAYS.

The project moved through four distinct phases. Each produced something concrete, and each informed what came next.

Phase 2 — Design System Before Pages. I built the full token system first: the six-colour palette, the Cormorant Garamond / DM Sans type pairing, the spacing scale, three motion curves, and the component library. Only then did I touch a page layout. Designing without a system produces pages that look different from each other — for a luxury brand, incoherence is fatal.

03 — Key Decisions

WHY THIS, NOT THAT.

Product page layout — 3 options explored, editorial split chosen
01
Slide-Out Cart Drawer

Instead of navigating to a /cart page, I built a slide-out drawer from the right. The product page stays visible in context. Users see their cart, confirm the item, and continue browsing — without breaking the experience. This is the standard on premium e-commerce for exactly this reason.

02
Two Slate Tokens, Not One

No single grey value can pass 4.5:1 on both light and dark backgrounds simultaneously — the maths makes it impossible. Solution: —slate (#5A5A53, 6.27:1 on linen) and —slate-dark (#A8A89E, for dark sections). Two tokens, both passing, both visually consistent. This is what separates a design system from a colour list.

03
Three Motion Curves

Curtain/modal: cubic-bezier(0.76, 0, 0.24, 1) — dramatic, like moving heavy furniture. Scroll/hover: cubic-bezier(0.4, 0, 0.2, 1) — smooth but not soft. Cart bump: cubic-bezier(0.34, 1.56, 0.64, 1) — a spring overshoot. The only moment of playfulness in the motion system.

04
Event Delegation Over 79 Handlers

The initial build used inline onclick attributes on every interactive element — 79 in total. The refactor replaced all 79 with a single delegated click listener on document, using data-action attributes. The HTML became clean markup. The JS became auditable. A senior engineer reviewing this code would see the difference immediately.

05 — Reflection

WHAT I'D DO DIFFERENTLY.

Start with accessibility constraints, not add them at the end. The contrast failures affected 57 CSS rules because I chose the palette for aesthetics and audited afterwards. If I had run the contrast calculations before committing to the tokens, I would have chosen values that passed from day one — and the design would have looked essentially identical. The lesson: accessibility is a design input, not a QA step.

Build the event system first. The 79 inline onclick handlers were added incrementally as the prototype grew. Starting with a delegation pattern from the first interactive element would have produced a cleaner codebase with no extra effort.

What I'd add with more time: Shopify Storefront API integration for live product data. A dark mode (the colour system is structured for it). Usability testing with 5 participants on the product-to-checkout flow. A proper checkout flow — the modal is a placeholder. Image zoom on the product gallery.
